# Clotho V1 迁移顺序建议

**版本**: 0.1.0
**日期**: 2026-04-08
**状态**: Draft
**作者**: Codex

---

## 1. 文档目的

本文档定义从现有原型过渡到 `11_v1_app/` 的推荐迁移顺序。

目标不是“把旧项目搬家”，而是：

- 保留高价值资产
- 避免继承原型期技术债
- 让 V1 主链路尽快形成可运行闭环

## 2. 总体结论

推荐采用 **选择性迁移 + 新主干重建**：

1. 冻结 `08_demo/` 与 `09_mvp/` 的职责
2. 建立 `11_v1_app/` 骨架
3. 只迁移低风险、高价值资产
4. 核心运行链路按 V1 规格重新实现

不建议采用：

- 在 `08_demo/` 基础上继续堆业务逻辑
- 在 `09_mvp/` 基础上持续大改
- 先整体复制旧目录，再边实现边清理

## 3. 迁移来源判定

### 3.1 可直接迁移

满足以下条件的内容可优先迁移：

- 与业务状态无强耦合
- 与数据库无直接耦合
- 与现行协议无明显漂移
- 已经具有可复用的视觉或结构价值

### 3.2 只可参考，不可直接继承

满足以下条件的内容应仅作为参考：

- 原型期临时装配
- in-memory 持久化实现
- 直接依赖 mock 数据流程
- 与当前 `00_active_specs/` 存在边界漂移

## 4. 迁移矩阵

| 来源 | 内容类型 | 建议动作 | 说明 |
|------|------|------|------|
| `08_demo/lib/theme/` | 主题与设计令牌 | 迁移 | 属于低耦合高价值资产 |
| `08_demo/lib/widgets/` | 纯展示组件 | 选择性迁移 | 需先剥离 mock 依赖 |
| `08_demo/lib/screens/home_screen.dart` | 页面原型 | 重写 | 可借布局，不应直接继承状态逻辑 |
| `09_mvp/lib/jacquard/` | 编排原型 | 参考重写 | 保留命名与流程思路，不直接沿用实现 |
| `09_mvp/lib/mnemosyne/repositories/in_memory_*` | 内存仓库 | 丢弃 | 与 V1 持久化目标冲突 |
| `09_mvp/lib/stage/providers/` | 原型装配层 | 丢弃 | 职责边界不适合作为正式基座 |
| `09_mvp/lib/jacquard/services/filament_parser.dart` | 协议解析原型 | 参考重写 | 需与 V1 冻结契约重新对齐 |
| `09_mvp/test/` | 测试思路 | 参考迁移 | 优先保留断言思路，而非直接复制测试体 |

## 5. 推荐迁移阶段

### 5.1 阶段 A: 冻结原型身份

目标：

- 明确 `08_demo/` 与 `09_mvp/` 不再作为正式实现主干
- 将实现目标转移到 `11_v1_app/`

动作：

- 在规划文档中固定 `11_v1_app/` 为 V1 主干
- 后续开发任务默认以 `11_v1_app/` 为目标目录

完成标志：

- 团队对三个目录职责达成一致

### 5.2 阶段 B: 建立空骨架

目标：

- 创建 `11_v1_app/` 最小可启动结构

动作：

- 创建 Flutter 应用根目录
- 建立 `bootstrap/`、`app/`、`shared/`、`presentation/`、`jacquard/`、`mnemosyne/`、`muse/`
- 建立最小测试目录

完成标志：

- 应用可以启动到空壳界面
- 目录职责已经固定

### 5.3 阶段 C: 先迁移视觉资产

目标：

- 让新主干尽快获得稳定的 UI 基础

动作：

- 从 `08_demo/` 迁移主题与 design tokens
- 迁移纯展示型基础组件
- 在新主干内重建聊天界面骨架

完成标志：

- `11_v1_app/` 具备基本可浏览界面
- UI 资产不再依赖旧 demo

### 5.4 阶段 D: 重建 Mnemosyne 最小骨架

目标：

- 建立 V1 所需最小持久化闭环

动作：

- 定义 SQLite 最小 DDL
- 实现 `sessions`、`turns`、`messages`、`active_states`、`state_oplogs`
- 建立 repository 接口与 SQLite 实现

完成标志：

- 可以创建、读取、删除 Session
- 可以保存并恢复最小历史与状态

### 5.5 阶段 E: 重建 Jacquard 主链路

目标：

- 建立真正可运行的主对话闭环

动作：

- 实现 Context 装载
- 实现最小 PromptBundle 构建
- 实现 Muse Raw Gateway 调用
- 按 V1 契约重写 Filament 解析
- 接入 State Updater

完成标志：

- 可以完成连续对话
- 可以解析 `<content>`
- 合法 `<state_update>` 可写入并恢复

### 5.6 阶段 F: UI 接线与恢复能力

目标：

- 让 Presentation 与新主干真正闭环

动作：

- UI 只通过适配器调用用例
- 接入流式显示
- 接入 Session 列表与恢复入口
- 接入只读状态查看器

完成标志：

- 重启后可恢复历史
- 流式输出稳定
- 错误场景有可见反馈

### 5.7 阶段 G: 清理原型耦合

目标：

- 确保 V1 不再依赖旧 demo 运行

动作：

- 清点 `11_v1_app/` 中所有来自旧目录的迁移内容
- 删除或替换残留的 mock 流程
- 清理临时桥接代码

完成标志：

- `11_v1_app/` 可独立构建与运行
- 旧目录仅保留参考价值

## 6. 优先级排序

推荐按以下顺序推进：

1. 目录骨架
2. 视觉资产
3. 持久化骨架
4. 主对话链路
5. 状态写回
6. 恢复与错误处理
7. 调试与验收

这个顺序的关键点是：

- 不让 UI 先和错误的数据层耦合
- 不让数据库先服务一个还没冻结的编排流
- 不让原型代码在新主干里继续扩散

## 7. 每阶段的复用策略

### 7.1 从 `08_demo/` 复用

优先复用：

- 主题
- 设计令牌
- 纯展示组件
- 页面布局视觉语言

复用方式：

- 复制后在 `11_v1_app/` 中重新归位
- 以新主干命名和边界为准进行调整

### 7.2 从 `09_mvp/` 参考

优先参考：

- 模块拆分思路
- 领域对象命名
- 原型阶段已验证过的流程顺序
- 测试断言思路

参考方式：

- 重新实现
- 对照当前冻结契约修正漂移点

## 8. 不应发生的迁移方式

以下迁移方式应明确避免：

- 整个 `08_demo/lib/` 直接复制到 `11_v1_app/lib/`
- 整个 `09_mvp/lib/` 直接复制到 `11_v1_app/lib/`
- 为了图快，先把 in-memory 仓库接上 UI 再说
- 在迁移期间让 UI 临时直接读取 SQLite

## 9. 迁移完成的定义

只有满足以下条件，才能认为迁移完成：

1. `11_v1_app/` 已经形成自洽的独立应用
2. `11_v1_app/` 不再依赖 `08_demo/` 或 `09_mvp/` 运行
3. V1 主链路可运行、可持久化、可恢复
4. 旧目录只作为参考，不再承担正式交付职责
