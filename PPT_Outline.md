### Slide 2: 现状与痛点 (Why Clotho?)
*   **现有方案的局限 (The "SillyTavern" Era)**
    *   **性能瓶颈**: 现有的许多 AI RPG 客户端（如 SillyTavern）主要基于 Web 技术栈构建。在处理成千上万条的长对话历史时，浏览器的内存管理和 DOM 渲染机制往往不堪重负，导致严重的内存泄漏、页面卡顿甚至崩溃，极大地破坏了用户的沉浸感。
    *   **逻辑混杂**: 为了快速迭代，许多项目采用了“意大利面条式”的代码结构，将 UI 渲染代码与复杂的业务逻辑深度耦合。这不仅使得代码难以维护和扩展，也导致了“牵一发而动全身”的脆弱性，任何微小的界面调整都可能破坏核心功能。
    *   **时空错乱**: 缺乏严谨的状态管理机制。在 RPG 体验中，用户经常进行“回溯 (Undo)”或“重绘 (Reroll)”操作。由于缺乏不可变的状态快照，这些操作往往会导致世界状态（如角色属性、物品栏）与对话历史不一致，造成“世界线崩坏”，使得长线叙事难以维系。
*   **Clotho 的破局之道**
    *   **Native First**: 我们摒弃了 Web 容器，全面拥抱 **Flutter** 生态。利用 Flutter 的 Skia/Impeller 图形引擎，我们实现了跨平台（Windows, Android 等）一致的 60fps+ 高性能原生体验，彻底解决了长列表渲染和内存管理问题。
    *   **Architecture**: 引入了严格的分层架构 (Layered Architecture)，将系统划分为表现层、编排层和数据层。通过单向数据流和明确的接口定义，彻底解耦了逻辑与表现，确保了系统的健壮性和可维护性。

### Slide 3: 核心哲学 (The Caesar Principle)
*   **核心理念**: 凯撒原则 (The Caesar Principle)
    *   *"Render unto Caesar the things that are Caesar's, and unto God the things that are God's."* (凯撒的归凯撒，上帝的归上帝)
*   **凯撒的领域 (Code's Domain)**
    *   **特性**: 100% 确定性、逻辑严谨、执行迅速。
    *   **职责**: 
        *   **逻辑判断**: 战斗伤害计算、技能判定、分支条件检查。
        *   **数值计算**: 经济系统、经验值增长、属性变化。
        *   **状态管理**: 维护角色卡、物品栏、世界变量的单一真实来源 (SSOT)。
        *   **工具调用**: 执行系统级操作、文件读写、API 请求。
    *   **意义**: 这些任务容不得半点模糊，必须由确定性的代码逻辑来严格把控，绝不能交给概率性的 LLM 去“猜测”。
*   **上帝的领域 (LLM's Domain)**
    *   **特性**: 概率性、创造性、语义丰富、模糊处理。
    *   **职责**: 
        *   **语义理解**: 解析用户的自然语言意图，理解上下文的隐含信息。
        *   **情感演绎**: 生成符合角色设定的对话，展现丰富的情感色彩。
        *   **剧情生成**: 在逻辑框架内进行创意写作，推动故事发展。
        *   **文本润色**: 优化输出文本的流畅度和文学性。
    *   **意义**: 这是 AI 的神性所在，也是它的核心价值。我们应让 LLM 专注于它最擅长的创造性工作，而不是让它去处理它不擅长的算术和逻辑。
*   **结论**: 拒绝 LLM 全权接管一切的懒惰设计，构建 **Hybrid Agency (混合代理)** 模式。让确定性的代码成为骨架，让概率性的 AI 成为血肉，两者结合，共同编织出既有逻辑深度又有情感温度的完美体验。

## 第二部分：架构篇 - 纺织隐喻 (The Architecture)

### Slide 4: 宏观架构全景 (The Three Pillars)
*   **核心隐喻体系 (Textile Metaphor)**
    Clotho 采用了独特的纺织隐喻来命名其核心子系统，象征着我们将数据、逻辑和表现编织在一起的过程。
    *   **Jacquard (提花机)**: **编排层**。它是系统的“大脑”与流水线。就像提花机控制经纬线的交织来编织图案一样，Jacquard 编排 Prompt 的组装、LLM 的调用以及结果的解析，确定的控制着生成的流程。
    *   **Mnemosyne (记忆女神)**: **数据层**。它是系统的“海马体”与上下文引擎。Mnemosyne 不仅存储数据，更负责管理多维度的记忆链条，为 AI 提供精准的上下文支持，确保它“记得”发生过的一切。
    *   **The Stage (舞台)**: **表现层**。它是纯粹的渲染界面。就像剧场舞台只负责呈现表演而不决定剧本一样，The Stage 只负责将数据可视化，并将用户的操作转化为意图发送给后台，它本身不包含业务逻辑。
*   **数据流设计**
    *   **单向数据流**: 遵循 MVU (Model-View-Update) 架构思想。UI 层（The Stage）严禁直接修改底层数据。用户的每一次点击、输入都会被转化为一个 **Intent (意图)**，发送给 Jacquard 处理。
    *   **状态投影**: 逻辑层处理完 Intent 后，会更新 Mnemosyne 中的状态，并通过 **Snapshot (快照)** 的形式将最新的状态树广播回 UI 层。这种设计确保了状态的可预测性和调试的便利性。

### Slide 5: Jacquard (编排层) - The Loom
*   **核心定义**: 确定性的 **Pipeline Runner** (流水线执行器)。Jacquard 不直接硬编码业务逻辑，而是作为一个通用的运行时环境，按顺序执行一系列插件化的任务。
*   **Skein (绞纱)**
    *   Jacquard 处理的核心数据对象不仅仅是字符串，而是一个名为 **Skein** 的异构容器。
    *   它结构化地管理着 System Prompt（系统指令）、World Info（世界设定）、User Input（用户输入）和 Chat History（对话历史）。
    *   支持 **深度注入 (Depth Injection)**，允许在历史记录的特定深度插入信息（如“在倒数第 3 条消息前插入当前场景描述”）。
    *   集成 **Jinja2 动态模板**，支持在 Prompt 中使用逻辑控制（如 `{% if hp < 10 %}...{% endif %}`），实现了 Prompt 的动态编程。
*   **流水线流程**
    1.  **Planner (Pre-Flash)**: 在生成前进行规划，决策本轮对话的焦点、目标和策略（如“是否触发随机事件？”）。
    2.  **Builder**: 根据配置和规划，从 Mnemosyne 获取数据，组装成结构化的 Skein 对象。
    3.  **Renderer**: 执行 Jinja2 模板渲染，将动态变量替换为实际文本，生成最终发给 LLM 的 Prompt。
    4.  **Invoker**: 调用 LLM API（支持流式输出），处理网络请求和重试逻辑。
    5.  **Parser**: 实时解析 LLM 输出的 Filament 协议，分离对话内容和指令数据。
    6.  **Updater**: 收集解析出的指令，向 Mnemosyne 提交状态变更请求，完成闭环。

### Slide 6: Mnemosyne (数据层) - The Memory
*   **超越数据库**: Mnemosyne 远不止是一个 SQLite 封装，它是 **动态上下文生成引擎**。它理解数据的语义，并能根据需要即时构建出最适合当前情境的上下文。
*   **多维链网结构**
    *   **History Chain (历史链)**: 记录线性的对话文本。这是人类阅读的故事线，也是 LLM 理解上下文的基础。
    *   **State Chain (状态链)**: 维护一棵 **VWD (Value with Description)** 状态树。每个节点不仅包含数值（Value），还包含语义描述（Description），例如 `HP: [80, "当前健康状况，低于 0 死亡"]`。这种结构让 LLM 能理解数值背后的含义。
    *   **Event Chain (事件链)**: 记录不可变的客观事实 (Immutable Facts)，如“击败了哥布林王”、“获得了圣剑”。这些记录构成了世界的“绝对真理”，不会随回溯而改变（除非显式重写）。
*   **Punchcards (穿孔卡)**
    *   这是一个基于 **Time Pointer (时间指针)** 的概念。Mnemosyne 可以根据任意时间点，动态计算并生成该时刻的完整世界快照 (Punchcard)。
    *   这使得 Clotho 能够实现无损的、像素级的 **时间回溯**。当你点击“撤销”时，整个世界瞬间回到之前的状态，所有的属性、物品、剧情进度都精确还原，绝无错乱。

### Slide 7: Presentation (表现层) - The Stage
*   **技术基石**: 基于 Google 的 **Flutter** 框架构建，利用其底层的 **Skia/Impeller** 渲染引擎，直接在 GPU 上绘制 UI，避开了 Web DOM 的性能开销，实现了如丝般顺滑的动画和滚动体验。
*   **Hybrid SDUI (混合驱动 UI)**
    为了兼顾高性能和扩展性，我们设计了双轨渲染架构：
    *   **Native Track (优先)**: 核心 UI 组件使用 **RFW (Remote Flutter Widgets)** 技术。这允许服务器（或本地逻辑层）下发二进制的 UI 描述数据，由客户端直接渲染为原生 Flutter 组件。这既保留了动态更新的能力，又拥有原生的极致性能。
    *   **Web Track (兜底)**: 对于社区制作的、极其复杂的自定义内容（如基于 HTML5 Canvas 的小游戏或特效），我们提供 WebView 容器作为兜底方案。通过 JS Bridge，这些 Web 内容依然可以与 Clotho 核心进行安全的交互。
*   **布局哲学**
    *   **Stage (舞台)**: 占据屏幕核心区域，专注于展示沉浸式的对话和剧情。设计上极简、无干扰，让用户仿佛置身于故事之中。
    *   **Control (控制台)**: 将复杂的参数配置、Lorebook 管理、状态检视器 (Inspector) 等功能收纳于侧边栏或抽屉中。它们在需要时触手可及，在不需要时隐形，不喧宾夺主。

## 第三部分：核心机制深潜 (Deep Dive)

### Slide 8: Filament 协议 - The Common Language
*   **定位**: 在自然语言（人类/LLM 的语言）与机器指令（代码的语言）之间，存在着巨大的模糊地带。Filament 协议就是为了消除这种模糊而生的通用交互语言。
*   **非对称设计 (Asymmetric Design)**
    我们认识到，LLM 的输入和输出有着截然不同的特性需求：
    *   **Input (Context) -> XML + YAML**: 输入给 LLM 的上下文需要高可读性且 Token 效率高。YAML 格式简洁紧凑，非常适合描述复杂的层级状态；XML 标签则清晰地界定了不同内容块（如 `<char_card>`, `<world_info>`）的边界，防止 LLM 混淆指令与数据。
    *   **Output (Instruction) -> XML + JSON**: LLM 输出的指令需要机器能够精确解析。JSON 具有严格的语法标准，易于校验和序列化，非常适合描述结构化的参数；XML 标签（如 `<function_call>`, `<state_update>`）则作为显式的意图标识，让解析器能快速定位指令。
*   **应用范畴**: Filament 协议不仅仅用于 LLM 交互，它贯穿了 Clotho 的全链路：从 Prompt 工程的构建，到 UI 层的渲染指令，再到底层状态的更新操作，全部统一在这一套协议标准之下，实现了系统内部语言的高度统一。

### Slide 9: 记忆与叙事 (Memory & Narrative)
*   **Turn-Centric 架构 (回合制叙事)**
    *   Clotho 将叙事的基本单元定义为“回合 (Turn)”，而不仅仅是单条消息。一个回合包含了用户的输入、系统的响应、状态的变更以及潜在的思维链。
    *   每个 Turn 对象内部整合了 **Summary (自动摘要)** 与 **Vector ID (向量索引)**。这意味着系统在进行 RAG（检索增强生成）时，不再是盲目地检索原始文本，而是检索经过压缩和语义提取的摘要，极大地提高了检索的准确性和效率。
*   **生命周期管理**
    *   **Pre-Flash (规划阶段)**: 在 LLM 生成文本之前，Planner 组件会先运行。它会根据当前情境，决定本轮对话的“聚光灯”打在哪里，检索哪些相关的记忆，甚至预设生成的策略（如“激进”或“保守”）。
    *   **Post-Flash (整合阶段)**: 在 LLM 生成文本之后，Mnemosyne 会在后台异步工作。它会对新的对话内容进行压缩，提取关键事实存入 Event Chain，生成角色的反思（Reflection），并将这些长期记忆归档。这个过程模仿了人类大脑在睡眠时整理记忆的机制。

## 第四部分：总结与展望 (Conclusion)

### Slide 10: 总结与展望
*   **项目价值**
    *   Clotho 的意义远超出一个高性能的 RPG 客户端。它实际上定义了一套 **AI 互动叙事的标准与规范**。我们提出的架构模式、协议标准和交互哲学，为未来 AI 游戏和互动小说的开发提供了可复用的范本。
*   **核心竞争力**
    *   **确定性**: 我们成功地用确定性的代码逻辑约束了 AI 的不稳定性，让 AI 变得可控、可预测，从而能够承载复杂的长线叙事。
    *   **高性能**: 通过 Native First 策略，我们让 RPG 体验回归了应有的流畅与沉浸，打破了 Web 技术栈的性能天花板。
    *   **可扩展**: 插件化的 Jacquard 架构与标准化的 Filament 协议，使得 Clotho 拥有极强的扩展性，社区可以轻松地为其开发新的功能模组和内容包。
*   **下一步计划**
    *   **完善 MVP Demo**: 尽快推出包含核心功能的演示版本，验证架构的可行性。
    *   **推进 Filament 协议标准化**: 将 Filament 协议开源并文档化，争取成为行业事实标准。
    *   **扩展社区插件生态**: 开发开发者工具包 (SDK)，鼓励社区为 Clotho 贡献更多样化的插件和剧本。
